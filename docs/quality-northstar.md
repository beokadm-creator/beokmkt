# 품질 North-Star — 운영·관리자·사용자 화면 개선 프롬프트

> 이 문서는 새 세션/에이전트에 그대로 넘겨 "최고 품질로 수렴"시키기 위한 작업 지침이다.
> 목적: 블로그 자동발행 시스템의 **콘텐츠·운영·관리자 UI·사용자 UI** 품질을 끝까지 끌어올린다.

---

## 0. 절대 원칙 (먼저 읽어라)

1. **품질은 "실제 산출물을 직접 읽어" 검증한다.** `node --check`/import 성공/`status: success`는 검증이 아니다.
   과거 네이버 글이 "발행 성공"으로 찍히고도 평문 쓰레기였고, glm-5.1이 한국어에 한자(`信信`, `几次`)를 섞는 결함은
   **출력을 눈으로 읽어야만** 잡혔다. 모든 품질 변경은 실제 1~2건 생성/렌더 후 사람이 읽고 확인한다.
2. **사용자가 지정하지 않은 것을 임의로 삭제·개조하지 마라.** 방향이 모호하면 묻는다.
3. **채널마다 품질 천장이 다르다(§4). 천장을 무시한 약속을 하지 마라.**
4. 새 사실·수치·사례를 지어내지 마라. 구조·가독성은 마음껏 개선하되 주장은 근거 기반.

## 1. 시스템 현실 (그라운딩)

- **파이프라인**: `blog_publisher/` Python + SQLite(`db/blog.db`). 상태머신:
  `draft → generating → factchecking → reviewing → reviewed → queued → publishing → published | needs_human | failed`.
  워커 간 직접 호출 없음 — 오직 `posts.status`로 협업. cron이 단계별 워커를 돌린다.
- **채널 4종**: 자체 블로그(Firebase, 신뢰 채널), 네이버(Node 워커 Playwright), 티스토리(동), 트위터/X.
- **생성 엔진**: 근거기반 — ①의도/키워드 ②자료수집 ③근거팩 ④개요 ⑤섹션 ⑥SEO. 모델 glm-5.1(z.ai).
- **관리자 UI**: `src/spa/` (React). 대시보드는 SQLite 파이프라인 모니터링(`/api/pipeline/stats`).
- **사용자 UI**: 공개 블로그 `src/spa/pages/PublicBlog*`, SSR은 `functions/ssr-template.mjs`.
- **현재 설정 주의**: `MIN_GROUNDING_RATIO=0.0`(grounding 게이트 사실상 꺼짐), `MODEL_SECTION=glm-5.1`.

## 2. 가장 큰 효과 순서 (로드맵)

### ✅ Phase A — 생성 깊이·구조 (완료, 실측 검증됨)
- `SECTION_SYSTEM`: "마크다운 최소·400~650자" → **600~1000자, 소제목/목록/굵게/표, 깊이, 순한글 강제**.
- `MAX_TOKENS_SECTION` 700→1500, 섹션 thinking ON.
- 한자 혼입 감지 재시도 + 최후 제거(`_count_hanzi`/`_strip_hanzi`).
- 실측: 400~650자 평문 → 1600자+ 구조화/순한글. **자체 블로그에 직접 반영됨.**

### ✅ Phase B — 티스토리/자체 블로그 디자인 (실제 티스토리 발행 검증 완료)
- 티스토리·자체 블로그는 **리치 HTML이 살아남는 채널**(네이버와 다름).
- `channel-rewriter.mjs`의 티스토리 가이드를 중복회피 중심에서 **품질 지향**으로 개편:
  핵심 요약, h2/h3, 목록, 번호 절차, 비교표, 체크리스트, blockquote 콜아웃을 요구하고 낚시성 톤 금지.
- `tistory-html-adapter.mjs`는 두 번째 AI 디자인 호출을 기본 비활성화(`TISTORY_AI_DESIGN=true`일 때만 사용).
  기본 경로는 결정론적 HTML 스타일러로 바꿔 발행 timeout 리스크를 낮춘다.
- 자체 블로그 `render/renderer.py`·`style.css`에 표 래퍼, 체크리스트, 콜아웃, beok CTA 조건을 추가.
- 실측:
  - 긴 원고(id=16) 티스토리 변환: 2,590자 / h2 6 / 목록 5 / 표 1 / 콜아웃 2 / 강조 17 / 금지톤 0 / 취소선 0.
  - 자체 블로그 모바일 렌더: summary-card, TOC, CTA 노출 / 오버플로 0.
  - 실제 티스토리 신규 발행(id=23 → `https://beoksolution.tistory.com/17`): published, 공개 URL 저장, RSS/공개 HTML 확인.
    공개 HTML 기준 h2 9 / h3 10 / 목록 16 / li 35 / 표 2 / blockquote 2 / 강조 13 / 금지톤 0 / 취소선 0.
- 실측 중 발견한 의미 반전 위험 문장("핵심은 단순히 ... 기능에 있습니다")은 `channel-rewriter.mjs` 게이트로 추가 차단.
- 2026-06-14 후속 보강:
  - 티스토리 발행 직전 `validateTistoryHtml()`로 글자수·소제목·목록·표/콜아웃/이미지/강조 구조를 검사한다.
  - 짧은 글/구조 붕괴 글은 `TISTORY_HTML_QUALITY_FAILED`로 발행하지 않고 격리한다.
  - 자체 블로그 beok CTA는 홈페이지 제작 문구가 아니라 학회 명찰 출력·현장 재발행 지원 문구로 분기한다.
  - `/blog/` 공개 인덱스는 홈페이지 구독 랜딩이 아니라 최신 글·운영 주제 중심의 콘텐츠 허브로 정리했다.

### ✅ Phase C — beok 이미지 기본 확장 (실제 자산 한계 내 완료)
- `beoksolution.com` 공개 사이트에서 실제 이미지 자산을 수집한 결과 현재 확인 가능한 자산은 `https://beoksolution.com/img/logo.png`와 `favicon.svg`뿐이다.
- 실제 제품 화면을 날조하지 않기 위해, 제품 스크린샷 대신 **브랜드 정보 카드 SVG 4종**을 자체 호스팅 자산으로 추가했다.
  - 운영 흐름, SEO 점검, 자동화 연결, 운영 체크리스트.
- `tools/image_bank.py`는 beok 글에서 h2 섹션 문맥에 맞춰 최대 3개 이미지를 중복 없이 삽입한다.
- 검증 기준: `https://beokmkt.web.app/assets/blog/beok/*.svg`가 `image/svg+xml`로 응답하고, 본문에 실제 `<img>`로 렌더되어야 한다.
- 남은 일: 서비스 화면, 관리자 화면, 예약/결제/알림톡/AI 자동화 등 **실제 제품 이미지 URL**이 추가되면 SVG 카드보다 우선하도록 확장.

### ◐ Phase D — 네이버 현실적 천장 (품질 게이트 적용, 지속 모니터링)
- SmartEditor는 inline CSS·div·CTA를 벗긴다. 페이스트 HTML로 "디자인"은 불가.
- 달성 가능 최대: **깨끗한 소제목·굵게·목록·이미지**. 진짜 디자인은 SE3 네이티브 컴포넌트 조작(대공사) — 비용 대비 보류 권장.
- 2026-06-14 실발행에서 SmartEditor 취소선 토글이 켜진 상태로 본문 전체가 `<strike>` 저장되는 문제를 확인했다.
- 워커는 입력 전 취소선 툴바를 끄고, 발행 후 공개 URL을 다시 조회해 금칙 톤·오타·보이는 취소선이 있으면 성공으로 인정하지 않는다.
- 검증: 네이버 id=31 공개 글은 금칙 톤 0, 보이는 취소선 0. 실패한 네이버 산출물은 DB에서 `needs_human`으로 격리.

## 3. 관리자 화면 품질 방향

현재 대시보드(`DashboardPage.tsx`)는 파이프라인 모니터링으로 재작성됨(KPI/퍼널/채널/needs_human/최근).
2026-06-14 후속으로 `needs_human/failed` 목록에 자동 재시도 가능 여부, 차단 사유, 공개 글 링크, 품질 지표가 노출된다. 다음 단계:

- **로컬 재큐잉 UX 완성**: 클라우드 대시보드는 로컬 SQLite 큐를 직접 재등록하지 못한다. 로컬 관리자/CLI 재큐잉 흐름을 더 명확히 분리한다.
- **품질 지표 확장**: 글당 글자수·이미지수·grounding_ratio는 노출됨. 다음은 구조검증 결과와 공개 URL 검증 결과를 상세에 붙인다.
- **채널 세션 헬스**: 네이버/티스토리 세션 만료 임박 경고를 대시보드 카드로(현재는 webhook 알림만).
- **발행 미리보기**: 발행 전 렌더 결과를 관리자가 눈으로 확인하는 패널(품질 게이트의 인간 확인).
- 원칙: 운영자가 "지금 무엇이 막혔고 무엇을 누르면 되는지"를 한 화면에서 알게 한다.

## 4. 사용자(공개 블로그) 화면 품질 방향

- **읽기 경험**: 타이포·행간·목차(TOC)·이미지 레이지로딩은 `render/`에 있음. `/blog/` 인덱스는 최신 글·운영 주제 중심으로 정리됨. 다음은 개별 글 모바일 여백/관련글/표 스타일을 더 세밀하게 검증한다.
- **SEO/공유**: JSON-LD(Article), OG/Twitter 메타 존재(`renderer.py`). 내부링크·관련글·빵부스러기(BreadcrumbList) 추가로 체류·색인 강화.
- **신뢰 요소**: 작성일·출처·브랜드 푸터. CTA는 강매 아닌 "유용함" 톤(브랜드 힌트 규칙 준수).
- 검증: 실제 발행글 1건을 공개 URL로 열어 모바일/PC에서 직접 본다.

## 5. 채널별 품질 천장 (약속할 때 지켜라)

| 채널 | 천장 | 핵심 레버 |
|------|------|-----------|
| 자체 블로그 | **높음** | 생성 프롬프트(완료) + renderer 디자인 |
| 티스토리 | **높음** | rewriter 프롬프트 품질화 + HTML 디자인 |
| 네이버 | **중간** | 깨끗한 구조·이미지까지. 디자인은 네이티브 컴포넌트(대공사) |
| 트위터/X | 낮음(요약) | 요약 카피 품질 |

---

## 6. 작업 방식 (이 프롬프트를 받은 에이전트에게)

1. 한 번에 한 Phase. 효과 큰 순서(B → C → D, 관리자/사용자 UI는 병행 가능).
2. 변경 후 **반드시 실제 생성/렌더 1~2건을 사람이 읽고** 통과 확인. 통과 전 "완료" 선언 금지.
3. 외부 플랫폼 카테고리/자산 등 **알 수 없는 정보는 추측하지 말고 사용자에게 묻는다**.
4. 삭제·대규모 개조는 사용자 승인 후. 마케팅에 필요한 기능(예: KAID Insight)은 건드리지 않는다.
