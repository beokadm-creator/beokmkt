# 블로그 자동 발행 시스템 — 재설계 문서

## 0. 문제 진단

지금 "제대로 작동하는 느낌이 안 든다"의 원인은 두 가지가 섞여 있다. 둘은 원인도 해법도 완전히 다르므로 분리해서 봐야 한다.

| 증상 | 진짜 원인 | 해법 영역 |
|---|---|---|
| 원고가 평평하고 반복적 | 한 번에 긴 글을 통째로 생성 + thinking 끄고 입력으로 분량 채우기 | 생성 파이프라인 |
| 발행이 들쭉날쭉하고 잘 깨짐 | 네이버/티스토리에 안정적 공식 API가 없음 → 브라우저 자동화 의존 | 발행 안정화 |

핵심 인식: **발행 불안정은 모델 문제가 거의 아니다.** 모델 등급을 올리든 내리든 네이버/티스토리 자동화의 깨짐은 바뀌지 않는다. 모델은 원고 품질에만 영향을 준다.

## 1. 전체 아키텍처 — 한 흐름을 끊어 3개의 독립 단계로

지금 가장 큰 구조적 문제는 "생성 → 발행"이 한 프로세스로 붙어 있어 한 곳이 실패하면 전체가 멈추는 것이다. `posts` 테이블 하나를 중심에 두고, 각 워커가 **상태만 바꾸게** 한다.

```
[생성 워커]  draft ──────────────▶ draft(본문 채워짐)
[검수 워커]  draft ──pass──▶ reviewed
                  └─fail──▶ draft(재생성 대상)
[스케줄러]   reviewed ──▶ queued (발행 시각 분산)
[발행 워커]  queued ─▶ publishing ─▶ published
                                  ├▶ queued (재시도, 백오프)
                                  ├▶ needs_human (수동 폴백)
                                  └▶ failed (치명)
```

워커는 서로를 직접 호출하지 않는다. 각자 cron으로 독립 실행되며 DB 상태만 본다.

이 구조의 효과:
- 발행 워커가 죽어도 생성·검수는 계속 재고를 쌓는다.
- 생성이 하루 실패해도 발행은 재고(reviewed)에서 나간다 → **주기적 안정 발행**의 핵심.
- 어디서 막혔는지 상태값만 보면 즉시 파악된다.

### 상태 정의

| status | 의미 | 다음 단계 |
|---|---|---|
| `draft` | 시드만 있거나 본문 생성됨 | 생성/검수 워커 |
| `generating` / `reviewing` | 워커가 점유 중(중복 처리 방지) | 임시 |
| `reviewed` | 검수 통과, 발행 대기 재고 | 스케줄러 |
| `queued` | 발행 큐에 등록(next_run_at 예약) | 발행 워커 |
| `publishing` | 발행 진행 중(점유) | 임시 |
| `published` | 발행 완료(url 보유) | 종료 |
| `needs_human` | 자동 재시도 소진 → 사람이 처리 | 수동 |
| `failed` | 복구 불가 | 종료 |

## 2. 원고 생성 파이프라인 — 근거기반 엔진

> **이 절의 토대는 [planning/05](./planning/05-근거기반-콘텐츠엔진-재설계.md)·[06](./planning/06-유형별-템플릿-명세.md)·[07](./planning/07-검색노출-SEO-최적화.md)으로 재정의되었다.**
> 핵심 전환: "LLM이 글을 지어낸다" → "LLM이 **수집된 근거**를 원고로 합성하고, **타깃 검색엔진에 맞춰 노출 최적화**한다."
> 근거 없는 생성은 모델 등급과 무관하게 공허하고, 엔진을 모르는 글은 노출되지 않는다는 게 근본 진단이다.
>
> **채널별 타깃 엔진(기획 07):** 네이버 블로그→네이버 검색(C-Rank·D.I.A.+), 티스토리·자체→구글(표준 SEO). 구글 SEO와 네이버 SEO는 다른 게임이라 채널별로 최적화한다.

원칙: **한 번에 다 쓰지 않는다 + 상상하지 않는다.** 단계를 쪼개되, 모든 구체 주장은 웹 리서치로 모은 근거팩에서 나온다.

```
① 의도/키워드    주제+유형 → 검색 의도·주키워드·하위질문         (LLM, 추론 ON)
②a SERP 분석    타깃 엔진(네이버 API/구글) 상위 노출 분석        (research, 노출용)
②b 자료 수집    일반 웹에서 본문 추출(사실 근거)                (research, 근거용)
③ 근거팩        수집 자료 → 사실/엔티티/커버리지타깃(출처 귀속)   (LLM 추출)
④ 근거기반 개요   커버리지 타깃 + 유형 템플릿으로 목차            (LLM, 추론 ON)
⑤ 근거기반 섹션   섹션별 근거만 사용해 합성                       (LLM, thinking ON·1500토큰·한자 재시도)
⑥ SEO 최적화     엔진별 제목/메타/태그/포맷(네이버 이미지마커)    (seo)
⑦ 사실검증       주장↔근거팩 대조, grounding_ratio 게이트         (factcheck)
⑧ 품질 검수      규칙 + 문체/구조 LLM 게이트                      (02)
```

노출용 SERP 분석(②a)과 근거용 사실 수집(②b)은 분리한다 — 사실은 엔진과 무관하고, 노출 구성은 타깃 엔진을 따른다.

지원 유형: **리뷰 / 정보성(하우투) / 니치 전문** — 유형별 필수 섹션은 06 문서.

### 왜 이렇게 하나

- 근거 없는 LLM 생성은 일반론·부정확·무차별하다. 웹 리서치로 **사실을 먼저 모으고** 그것을 합성하면 "제대로 된 글"의 토대가 생긴다.
- 검수가 두 겹이다: **사실성**(⑥ factcheck, 근거팩 대조)과 **문체/구조**(⑦ review). 둘은 다른 문제다.
- 긴 단일 출력은 뒤로 갈수록 무너진다 → 섹션 단위로 끊어 품질 유지.
- 모델 등급은 `config.py`의 `MODEL_OUTLINE / MODEL_SECTION / MODEL_REVIEW`에서만 바꾼다.

### 모델 등급은 어떻게 정하나 (감 대신 데이터)

검수 게이트를 먼저 만들고 **통과율로 판단한다.**

1. 현재 모델로 게이트 통과율 측정(기준선).
2. 한 단계 싼 모델로 교체 → 통과율 재측정.
3. 통과율이 유지되면 싼 모델이 정답. 떨어지면 원복.

단계별로 따로 실험할 수 있다(예: 본문만 낮추고 개요는 유지). 비용 대비 품질 곡선을 보고 최적점을 찾는다.

## 3. 발행 안정화 로직

네 가지 장치가 핵심이다.

### (1) 상태머신 + 원자적 점유

`claim(id, from, to)`는 "from 상태일 때만 to로 바꾼다"를 한 SQL UPDATE로 처리한다. 두 워커가 같은 글을 동시에 집어도 한쪽만 성공한다.

### (2) 멱등성

각 글에 `idem_key`(채널+해시, UNIQUE). 재시도가 같은 글을 두 번 발행하지 않게 막는다.

### (3) 재시도 + 지수 백오프 + 격리

```
실패 종류        처리
RetryableError   attempts+1, 백오프(1→2→4분…최대 1시간) 후 queued 복귀
  attempts 초과   needs_human (수동 폴백 큐) + 알림
FatalError       즉시 failed + 알림
```

자동화는 100%를 목표로 하지 않는다. **90% 자동 + 실패분 사람 폴백**이 운영상 훨씬 안정적이다.

### (4) 원고 재고 버퍼 + 발행 시각 분산

- **재고 버퍼**: `reviewed` 재고를 `STOCK_BUFFER_DAYS`치 유지. 생성이 멈춰도 발행은 계속된다.
- **시각 분산(지터)**: 스케줄러가 하루 목표량만 큐에 올리되 `next_run_at`을 무작위 간격으로 흩뿌린다. 한꺼번에 쏘지 않아 특히 네이버 스팸 필터를 피한다.

## 4. 채널별 전략

| 채널 | 방식 | 안정성 | 전략 |
|---|---|---|---|
| 자체 블로그 | REST API/DB 직접 | 높음 | 신뢰 채널. 실패율 낮음 |
| 네이버 | Playwright(SmartEditor) | 낮음 | storage_state 세션 재사용, 사람형 타이핑, 실패 시 사람 폴백 |
| 티스토리 | Playwright(에디터) | 낮음 | 네이버와 동일 구조, 셀렉터만 다름 |

### 네이버/티스토리 깨짐 줄이기

1. 로그인 세션을 `storage_state.json`으로 저장·재사용한다. 매번 로그인하면 차단 위험이 커진다.
2. 사람 같은 타이핑 딜레이(20~80ms) + 발행 시각 지터.
3. 셀렉터는 어댑터 상단 `SELECTORS` 한곳에 모은다. DOM이 바뀌면 거기만 고친다.
4. 단계별 검증(에디터 진입/발행 버튼 활성)에 실패하면 `RetryableError`로 던져 재시도 → 최종 실패 시 `needs_human`.
5. 어댑터는 `Publisher` 인터페이스만 지키면 된다. 발행 워커는 채널 내부를 모른다.

## 5. 운영 체계 (cron)

각 단계를 독립 cron으로 돌려 서로 격리한다.

```
*/30 * * * *  python run.py generate    # 재고 생성
*/30 * * * *  python run.py review      # 검수 게이트
0    9 * * *  python run.py schedule    # 매일 아침 발행 큐 편성
*/5  * * * *  python run.py publish      # 큐 소비(분산 발행)
```

권장 모니터링 지표:
- 상태별 건수(`reviewed` 재고가 버퍼 이하로 떨어지면 생성 늘리기)
- 검수 통과율(모델 등급 조정 신호)
- `needs_human` 누적량(자동화 깨짐 추세)
- 채널별 발행 성공률

## 6. 다음 단계 제안

1. `config.py`의 자격증명/모델명을 환경변수로 채우고 `python run.py init`으로 시작.
2. 네이버/티스토리는 최초 1회 수동 로그인 후 `storage_state`를 저장하는 작은 도구(`tools/login.py`)를 추가.
3. 검수 게이트 임계값(`MIN_BODY_LEN`, `MAX_DUP_RATIO`, `MIN_REVIEW_SCORE`)을 실제 글로 1~2주 튜닝.
4. 통과율이 안정되면 본문 모델부터 한 단계씩 낮춰 비용 최적점 탐색.
5. `_notify()`를 슬랙/이메일에 연결해 `needs_human`·`failed`를 즉시 인지.
