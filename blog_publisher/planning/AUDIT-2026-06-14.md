## CRITICAL

### [CRITICAL] A1 — save_article가 NOT NULL 컬럼에 NULL을 써 모든 생성이 실패
**파일:** `db/db.py:188-195` (save_article), `db/schema.sql:33` (next_run_at NOT NULL)

**현상:** `next_run_at`은 `TEXT NOT NULL`인데 `save_article()`이 `next_run_at = NULL`로 UPDATE한다. 신규 SQLite로 재현한 결과:
```
save_article FAILED: IntegrityError NOT NULL constraint failed: posts.next_run_at
```
생성 워커는 `compose_article`로 본문을 만든 뒤 `save_article`에서 예외 → `run_once`가 `requeue_draft`로 되돌림 → 본문 유실 → `GENERATE_MAX_ATTEMPTS(5)` 소진 후 전부 `needs_human`.

**영향:** **신규 글이 한 건도 영속되지 않음.** 파이프라인의 첫 단계가 죽어 그 뒤(factcheck/review/publish)가 전부 굶는다. 실DB의 published 5건은 이 변경 이전 산출물이며, 지금 `run.py generate`를 돌리면 전건 실패한다.

**수정:** `next_run_at`을 NULL이 아니라 현재시각으로 설정(컬럼 마이그레이션 불필요).
```diff
--- a/db/db.py
+++ b/db/db.py
@@ def save_article(post_id: int, title: str, meta_desc: str, body: str) -> None:
     with connect() as conn:
         conn.execute(
             "UPDATE posts SET title = ?, meta_desc = ?, body = ?, status = 'draft', "
-            "attempts = 0, next_run_at = NULL, last_error = NULL, updated_at = ? WHERE id = ?",
-            (title, meta_desc, body, _iso(_utcnow()), post_id),
+            "attempts = 0, next_run_at = ?, last_error = NULL, updated_at = ? WHERE id = ?",
+            (title, meta_desc, body, _iso(_utcnow()), _iso(_utcnow()), post_id),
         )
```
(대안: 스키마에서 `next_run_at`을 nullable로 바꾸고 기존 DB를 `ALTER`. 운영 DB가 이미 NOT NULL이므로 위의 값-설정 방식이 더 안전.)

## HIGH

### [HIGH] D3 — 티스토리 세션 만료 오류코드 불일치 → 인증만료가 무한 재시도로 오분류
**파일:** `publishers/tistory.py:60`, `executors/naver-blog-worker/tistory-client.mjs:20,44`

**현상:** 워커는 세션 만료 시 `TISTORY_LOGIN_REQUIRED` / `TISTORY_NOT_AUTHED` 코드를 반환한다. 그러나 Python은
```python
if code in ("LOGIN_REQUIRED", "AUTH_REQUIRED"):  # ← TISTORY_* 가 없음
    raise FatalError(...)
raise RetryableError(...)
```
로 판정해, 티스토리 세션 만료를 **Fatal이 아닌 Retryable**로 처리한다. (네이버는 `LOGIN_REQUIRED`로 일치해 정상.)

**영향:** 티스토리 세션이 만료되면 "재인증 필요"라는 명확한 Fatal 신호 대신 **백오프 재시도를 max_attempts까지 낭비**한 뒤에야 needs_human이 된다. 운영자는 원인(재인증)을 즉시 알 수 없고, 그동안 티스토리 발행이 막힌다.

**수정:**
```diff
--- a/publishers/tistory.py
+++ b/publishers/tistory.py
@@
-        if code in ("LOGIN_REQUIRED", "AUTH_REQUIRED"):
+        if code in ("LOGIN_REQUIRED", "AUTH_REQUIRED",
+                    "TISTORY_LOGIN_REQUIRED", "TISTORY_NOT_AUTHED"):
             raise FatalError(
                 f"티스토리 세션 만료. executors/naver-blog-worker 에서 "
                 f"npm run tistory-auth 으로 재인증 후 워커를 재시작하세요."
             )
```

### [HIGH] 멱등성 부재 — 발행 부분성공 후 재시도 시 네이버/티스토리 중복발행
**파일:** `pipeline/publish.py:33-63`, `executors/naver-blog-worker/index.mjs:192-200`

**현상:** `idem_key`는 posts 테이블 UNIQUE 컬럼일 뿐, **워커는 idem_key를 받지도, 검사하지도 않는다.** 워커가 실제로 글을 올린 뒤 URL 캡처/네트워크 단계에서 예외가 나면(`capturePublishedUrl` 실패 등) Python은 `RetryableError`/`Exception`으로 받아 `requeue` → 다음 주기에 **같은 글을 다시 발행**한다. 네이버/티스토리에 동일 글이 2건 게시될 수 있다.

**영향:** 중복 게시 → 유사문서 필터 패널티 + 수동 정리 비용. 자동화의 신뢰성 핵심 결함.

**수정 방향:** 발행 요청에 `idem_key`(또는 post_id)를 실어 보내고, 워커가 최근 발행 로그(파일/메모리)와 대조해 동일 키 재요청 시 기존 URL을 반환하도록 가드. 최소 조치로는 Python에서 "URL 미회수 성공"과 "실패"를 구분해, 본문 입력 이후 단계의 예외는 재시도하지 않고 `needs_human`으로 보내 수동 확인.
