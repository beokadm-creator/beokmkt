## A. 한 줄 목표

> **사람이 읽기 좋고 구글·네이버에 잘 노출되는 "제대로 된 글"을, 웹 리서치 근거에 기반해
> 안정적으로·주기적으로 자동 생성하고 채널별로 발행한다.**

---

## B. 근본 방향 (왜 이렇게 가는가)

1. **상상하지 않고 근거로 쓴다.** AI 글이 공허한 진짜 원인은 모델 등급이 아니라 *근거의 부재*다.
   → 웹 리서치로 사실을 먼저 모으고(근거팩), 그것을 합성해 글을 쓴다. (기획 05)
2. **검색 노출이 관건이고, 엔진마다 게임이 다르다.** 구글 SEO ≠ 네이버 SEO.
   → 채널별 타깃 엔진 분리: 네이버 블로그→네이버(C-Rank·D.I.A.+), 티스토리·자체→구글(표준 SEO). (기획 07)
3. **발행 불안정은 모델이 아니라 채널 구조 문제.** 네이버/티스토리는 공식 API가 없다.
   → 상태머신·재시도·멱등성으로 안정화하고, 자동화 실패는 "사람 폴백"으로 흡수. (기획 03·04)
4. **품질은 감이 아니라 게이트로 강제한다.** 사실성(grounding)과 문체/구조를 분리 검증. (기획 02·05)
5. **주기적 안정 발행은 재고 버퍼 + 시각 분산으로 만든다.** (기획 03)

## C. 전체 파이프라인 (end-to-end)

```
시드(주제 + 유형 + 채널)
 → ① 검색 의도·키워드 도출
 → ②a 타깃 엔진 SERP 분석(네이버/구글)   ②b 일반 웹 사실 수집
 → ③ 근거팩(사실+출처 귀속 + 커버리지 타깃)
 → ④ 근거기반 개요(유형 템플릿: 리뷰/하우투/니치)
 → ⑤ 근거기반 섹션 작성(근거 안에서만)
 → ⑥ 엔진별 SEO 최적화(제목/메타/태그 / 네이버 이미지마커)
 → ⑦ 사실검증 게이트(grounding_ratio)
 → ⑧ 품질 검수(규칙 + 문체/구조)
 → ⑨ 발행 스케줄(재고 버퍼·시각 분산·윈도우)
 → ⑩ 채널 발행(자체=렌더 HTML / 네이버·티스토리=Playwright + 폴백)
```

부가 입력 경로:
- **URL 재작성**(기획 10): 원문 URL → 참고자료화 → 대폭 재구성 → 유사도 게이트 → 위 ⑦부터 합류.
- **영문 번역**(기획 11): 발행 가능한 ko 글 → 현지화 번역 + 영문 SEO → en 글로 발행.

## D. 상태머신 (운영의 중심)

`posts.status`: `draft → (generating) → factchecking → reviewing → reviewed → queued → publishing → published`
실패 분기: `needs_human`(자동 재시도 소진, 수동 발행) / `failed`(치명).
모든 워커는 서로 직접 호출하지 않고 **상태만 보고** 독립 cron으로 동작 → 한 단계가 죽어도 전체가 안 멈춘다.

## F. 운영 방식 (cron)

```
*/30 * * * *  generate     # 근거기반 생성(재고 적재)
*/15 * * * *  factcheck     # 사실검증
*/30 * * * *  review        # 품질 검수
0    9 * * *  schedule      # 발행 큐 편성(09~21시 분산)
*/5  * * * *  publish        # 큐 소비
```

핵심 KPI: 검수 통과율, 평균 grounding, `reviewed` 재고량, `needs_human` 누적, 채널별 발행 성공률.
모델 등급은 **통과율 데이터로** 결정한다(감 금지, 02 §3.4 / `tools/measure_passrate.py`).
