# GOAL PROMPT — 블로그 자동 발행 시스템 (닫는 프롬프트)

> 이 문서는 전체 기획(00~12)을 하나로 닫는 **최종 목표 프롬프트**다.
> 시스템·프롬프트·운영 결정은 모두 이 기준을 따른다. 변경 시 `CHANGELOG.md`에 기록한다.

---

## MISSION (목표 — 한 문장으로 고정)

**웹 리서치 근거에 기반해, 사람이 읽기 좋고 구글·네이버에 잘 노출되는 "제대로 된 글"을
안정적으로·주기적으로 자동 생성하여 채널별(자체·네이버·티스토리, ko/en)로 발행한다.**

성공의 1차 기준은 **파이프라인이 끊김 없이 가동되는 것**, 2차 기준은 **노출·품질**이다.

---

## SUCCESS CRITERIA ("제대로 된 글"의 정의 — 발행 가능 조건)

1. **의도 충족** — 타깃 검색 의도의 핵심 질문(coverage_targets)을 빠짐없이 답한다.
2. **사실 정확** — 모든 구체 주장(수치·고유명사·날짜·비교)에 근거팩 출처가 매칭된다. grounding ≥ 0.9.
3. **독창성** — 단일 출처 복붙이 아니라 복수 출처를 종합하고 관점을 더한다. 재작성은 원문 유사도 < 0.3.
4. **노출 구조** — 타깃 엔진(채널별)에 맞는 제목·메타·태그·구조. 과최적화 금지.
5. **자연스러운 한국어/영어** — AI 특유의 균일·반복·과도한 헤지 없음.
6. **유형 필수요소** — 리뷰/하우투/니치 각 유형의 필수 섹션 충족.

위 6개를 사실검증·품질검수 게이트로 강제한다. 하나라도 미달이면 발행하지 않는다.

---

## PIPELINE (고정 흐름 — 각 단계는 비협상)

```
시드(주제+유형+채널)
 → ① 의도/키워드        검색 의도·주키워드·하위질문
 → ②a 타깃 엔진 SERP     네이버 블로그→네이버 API / 티스토리·자체→구글
 → ②b 사실 수집          일반 웹(근거용). SERP 분석과 분리.
 → ③ 근거팩             사실+출처 귀속 + coverage_targets. 출처 없는 사실은 버린다.
 → ④ 근거기반 개요        커버리지 + 유형 템플릿. 상상으로 목차 만들지 않는다.
 → ⑤ 근거기반 섹션        해당 섹션 근거 안에서만 합성. 근거 밖 구체 주장 금지.
 → ⑥ 엔진별 SEO          제목/메타/태그(+네이버 이미지마커). 내용 왜곡 금지.
 → ⑦ 사실검증            주장↔근거팩 대조. grounding < 0.9면 재작성.
 → ⑧ 품질검수            규칙 + 문체/구조. 미달이면 draft 복귀.
 → ⑨ 스케줄              재고 버퍼 + 시각 분산(09~21시).
 → ⑩ 발행               자체=렌더 HTML / 네이버·티스토리=Playwright(+사람 폴백).
```

부가 경로: **URL 재작성**(원문=참고자료, 유사도·출처 가드레일) / **영문 번역**(현지화+영문 SEO) → ⑦부터 합류.

---

## NON-NEGOTIABLES (불변 원칙)

- **상상 금지.** 근거 없는 구체 주장은 만들지 않는다. 모든 사실은 출처에서 온다.
- **엔진은 채널이 정한다.** 구글 SEO와 네이버 SEO를 섞지 않는다.
- **발행은 100%를 노리지 않는다.** 네이버/티스토리 실패는 재시도→사람 폴백으로 흡수, 흐름을 멈추지 않는다.
- **저작권/중복 회피.** 재작성은 대폭 변형 + 출처 표기 + 유사도 검증. 스핀/복제 발행 금지.
- **품질은 게이트로 강제.** 모델 등급은 감이 아니라 통과율 데이터로 정한다.
- **구조화 출력은 JSON only.** chat_json + 견고 파서로 가동 신뢰성을 지킨다.
- **워커는 상태만 보고 독립 동작.** 한 단계가 죽어도 전체는 멈추지 않는다.

---

## OPERATING (가동 규칙)

- **AI 엔진**: 기존 GLM 사용. `config.py`의 `LLM_API_KEY`/`LLM_BASE_URL`/`MODEL_*`만 기존 값으로 지정.
- **cron**: generate(30m) · factcheck(15m) · review(30m) · schedule(매일 09시) · publish(5m).
- **가동 점검**: `python run.py selftest`(키 없이 전 파이프라인 PASS 확인) → 실키 연결 → 단건 `generate`.
- **KPI**: 검수 통과율 · 평균 grounding · reviewed 재고량 · needs_human 누적 · 채널별 발행 성공률.

## DONE = 

`python run.py selftest`가 PASS하고, 실제 GLM·검색·채널 키를 연결한 상태에서
한 시드가 ①~⑩을 통과해 **사람이 읽기 좋고 타깃 엔진에 노출되는 글로 발행**되며,
이 흐름이 cron으로 **끊김 없이 주기 반복**되면 목표 달성이다.

> 추적 규칙: 이 프롬프트 → 해당 기획 문서(00~12) → 코드(`llm/prompts.py` 등) 순.
> 모든 변경은 `planning/CHANGELOG.md`에 "대상/변경/이유"로 남긴다.
