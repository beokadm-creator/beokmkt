## 1. 문제의 근본 (왜 글이 공허한가)

기존 생성은 `주제 → (LLM이 상상으로) 개요 → 섹션 → 문체 검수`였다. 모델이 **아무 근거 없이 지어내는 구조**라서, 모델 등급·thinking 여부와 무관하게 결과가 일반론적이고 부정확하며 차별성이 없다. "제대로 된 글"이 안 나오는 진짜 원인은 모델이 아니라 **근거(grounding)의 부재**다.

근본 전환: **"LLM이 글을 지어낸다" → "LLM이 수집된 자료를 원고로 합성한다."**
검수도 문체가 아니라 **사실성**을 본다.

## 2. "제대로 된 글"의 정의 (측정 가능하게)

발행 가능한 글은 아래 6개를 모두 만족한다.

1. **의도 충족**: 타깃 검색 의도의 핵심 질문(coverage_targets)을 빠짐없이 답한다.
2. **사실 정확성**: 모든 구체적 주장(수치·고유명사·날짜·비교)에 근거팩 출처가 매칭된다. 근거 없는 구체 주장 0건.
3. **독창성**: 단일 출처 복붙이 아니라 복수 출처를 종합하고 관점/정리를 더한다.
4. **구조/SEO**: 제목·H2 구조·메타설명·요약이 검색 의도에 맞게 구성된다.
5. **자연스러운 한국어**: AI 특유의 균일·중복·과도한 헤지 표현이 없다.
6. **유형 필수요소**: 유형별 필수 구성요소(06 문서)를 충족한다. 예) 리뷰=스펙·장단점·결론.

이 6개가 검수 게이트(02)와 사실검증 게이트(본문 §6)의 판정 기준이 된다.

## 3. 새 파이프라인 (엔진 전체)

```
주제(+유형)
 → ① 의도·키워드 도출      LLM, 짧게         : 검색 의도 + 주 키워드 + 보조 키워드 + 하위질문
 → ② 자료 수집            SearchProvider     : SERP 상위 N + 본문 추출 + (PAA/연관질문)
 → ③ 근거팩 구성          LLM 추출, 출처귀속  : 검증된 사실/통계/엔티티 + 출처URL + 커버리지 타깃
 → ④ 근거기반 개요         LLM               : coverage_targets로 목차 구성(상상 금지)
 → ⑤ 근거기반 섹션 작성    LLM, 섹션별        : 해당 섹션 근거만 사용, 출처 인용
 → ⑥ 사실검증 게이트       LLM               : 주장↔근거팩 대조, 미근거 주장 제거/탈락
 → ⑦ 규칙+품질 검수        02 게이트          : 길이/중복/구조/자연스러움
 → ⑧ 발행                03/04
```

①~③이 추가된 **근거 계층**이고, ④~⑥이 근거에 묶인 생성/검증이다. 이게 토대 변화의 핵심.

## 4. 자료 수집 계층 (research)

### 4.1 SearchProvider 인터페이스 (공급자 교체 가능)

```python
class SearchProvider(Protocol):
    def search(self, query: str, k: int = 10) -> list[SearchResult]: ...   # title,url,snippet
    def fetch(self, url: str) -> str: ...                                   # 본문 텍스트
```

- 구현체는 환경에 맞게 교체(예: Tavily/SerpAPI/Bing/Google CSE + readability 추출).
- 출처 신뢰도 필터: 도메인 allowlist/blocklist, 광고·스팸 도메인 제외(설정).
- 과수집 방지: 상위 N개만, 본문 길이 상한, 호출 타임아웃/재시도.

### 4.2 수집 절차

1. ①에서 나온 주 키워드 + 하위질문들을 쿼리로 SERP 조회.
2. 상위 결과에서 신뢰 도메인 위주로 본문 추출.
3. 경쟁 글의 H2/소제목·PAA를 모아 **SERP 갭 분석** → 우리가 반드시 다룰 coverage_targets 산출(경쟁글이 답하는 질문 + 빠뜨린 질문).

## 5. 근거팩 (EvidencePack) 스키마

```json
{
  "topic": "...", "content_type": "review|howto|niche",
  "intent": "검색 의도 한 줄",
  "primary_keyword": "...", "secondary_keywords": ["..."],
  "coverage_targets": ["반드시 답할 질문/하위주제", "..."],
  "facts": [
    {"id": "f1", "statement": "구체 사실/수치", "source_url": "...", "source_title": "...", "confidence": "high|med|low"}
  ],
  "entities": [{"name": "...", "type": "제품/인물/기관/개념", "note": "...", "source_url": "..."}],
  "sources": [{"url": "...", "title": "...", "trust": "high|med|low"}]
}
```

- 모든 fact는 **출처 귀속**(source_url 필수). 출처 없으면 fact가 아님.
- LLM이 본문에서 사실을 추출할 때, 추출 텍스트 범위 밖의 내용을 지어내지 않도록 "주어진 본문에서만" 제약.

## 6. 사실검증 게이트 (factcheck)

발행 전 필수. 02 검수와 별개로 **사실성**만 본다.

1. 초안에서 원자적 사실 주장 추출(LLM).
2. 각 주장을 근거팩 facts와 대조(LLM): 지지/부분지지/미지지.
3. **미지지 구체 주장**(수치·고유명사 등)은 제거 또는 약화. 일반 상식 수준은 허용.
4. grounding_ratio = 지지 주장 / 전체 구체 주장. `MIN_GROUNDING_RATIO`(기본 0.9) 미만이면 탈락 → 재작성/draft 복귀.

## 7. 골격 반영 사항 (구현 대상)

- `research/`: `provider.py`(인터페이스+스텁), `collect.py`(수집), `evidence.py`(근거팩 빌더).
- `db/schema.sql`: posts에 `content_type, intent, keywords, evidence(JSON), sources(JSON), grounding_ratio` 추가.
- `pipeline/generate.py`: 상상 생성 제거 → 의도도출→리서치→근거팩→근거기반 개요/섹션.
- `pipeline/factcheck.py`: 사실검증 게이트 신규.
- `llm/prompts.py`: 의도도출/사실추출/근거기반 작성/사실검증 프롬프트 + 유형별 템플릿(06).
- `config.py`: 검색 공급자 키, 수집 상한, `MIN_GROUNDING_RATIO`.

(실제 수정은 CHANGELOG 기록)
