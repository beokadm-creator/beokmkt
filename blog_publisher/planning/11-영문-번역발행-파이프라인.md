# [11] 영문 번역 발행 파이프라인

- 상태: 확정
- 작성일: 2026-06-14
- 관련 골격: `pipeline/translate.py`, `llm/prompts.py`, `config.py`, `db/schema.sql`, `publishers/*`
- 선행 문서: [05](./05-근거기반-콘텐츠엔진-재설계.md), [07](./07-검색노출-SEO-최적화.md), [09](./09-자체블로그-디자인-템플릿.md)

## 1. 배경

한국어로 만든(또는 발행 대기) 글을 영문으로도 발행한다. 단순 직역이 아니라 **현지화(영어권 독자에 맞는 자연스러운 표현·맥락)** + **영문 SEO**가 필요하다.

## 2. 목표

- 원문(ko) → 자연스러운 영문 글(en) 생성. 직역체·어색함 제거.
- 영문용 제목/메타/키워드(영어권 검색 의도 반영).
- 원문과 번역본을 연결(`translated_from`), 영문 채널/로케일로 발행.
- hreflang 등 다국어 SEO 신호(자체 블로그).

## 3. 설계 / 결정

### 3.1 번역 단계

- 입력: 원문 글(title, body, tags, meta).
- LLM 번역: "직역 금지, 의미·뉘앙스 보존, 영어권 자연스러움". 마크다운 구조(H2 등) 유지.
- 고유명사/단위/통화/날짜 현지화(예: 원→USD 환산은 하지 않고 표기 보존하되 설명).
- 길이·정보 보존(누락 금지).

### 3.2 영문 SEO

- 번역 후 별도 SEO 단계(기획 07 구글 최적화 재사용): en 제목/메타/태그 재산출.
- 영어권 검색 의도가 한국과 다를 수 있으므로 키워드는 재도출(직역 키워드 지양).

### 3.3 발행/연결

- 새 post 생성: `locale='en'`, `translated_from=<원문 id>`, 채널은 영문 발행 대상(자체 영문 블로그 등).
- 자체 블로그: 렌더러가 `lang="en"` + (후속)hreflang.
- 번역본도 동일 게이트(품질 검수)·발행 흐름을 탄다(사실검증은 원문에서 이미 통과 → 생략 가능).

### 3.4 트리거

- 원문이 `reviewed`(또는 published)된 뒤 번역 생성. 모든 글을 번역할지/선별할지는 설정.

## 4. 골격 반영

- `db/schema.sql`: `locale`, `translated_from`(이미 추가).
- `pipeline/translate.py`: `translate_post(post, target_channel)` → en draft 생성.
- `llm/prompts.py`: `TRANSLATE_EN` 프롬프트.
- `config.py`: `TRANSLATE_ENABLED`, `EN_CHANNEL`(영문 발행 채널), 번역 모델.
- `run.py`: `translate <post_id>` 명령.
- 렌더러: `lang` 반영(이미 지원).

## 5. 미해결 / 다음 단계

- hreflang 상호 링크(ko↔en) 자동화.
- 영어권 SERP 분석에 별도 검색(영문 키워드) 적용.
- 다른 언어 확장(ja/zh) 시 동일 패턴 일반화.
- 번역 품질 검수(역번역 스팟체크 등).
