# [03] 발행 스케줄링 & 안정화 정책

- 상태: 확정
- 작성일: 2026-06-14
- 관련 골격: `pipeline/schedule_publish.py`, `pipeline/publish.py`, `db/db.py`, `config.py`
- 선행 문서: [DESIGN.md](../DESIGN.md) §3, §5

## 1. 배경 / 문제

발행이 들쭉날쭉하고 한 번 실패하면 흐름이 끊긴다. "주기적·안정적 발행"을 보장하는 스케줄링·재시도·격리 정책을 확정한다. 발행 불안정의 본질은 모델이 아니라 채널 구조(네이버/티스토리 API 부재)임을 전제로, 자동화 실패를 정상 흐름의 일부로 설계한다.

## 2. 목표 (end goal)

- 생성/검수가 하루 멈춰도 발행은 끊기지 않는다(재고 버퍼).
- 매일 목표 건수가 사람처럼 분산되어 발행된다(스팸 필터 회피).
- 실패는 자동 재시도 후, 소진되면 사람에게 위임된다(흐름 비차단).
- 어디서 막혔는지 상태값으로 즉시 보인다.

## 3. 설계 / 결정

### 3.1 재고 버퍼

- `reviewed` 상태 재고를 `STOCK_BUFFER_DAYS`(기본 3)일치 유지.
- 생성 워커는 "재고가 버퍼 미만일 때" 더 적극적으로 시드를 소비(후속: 동적 생성량).
- 효과: 생성/LLM 장애가 발행에 즉시 전파되지 않음.

### 3.2 발행 시각 분산

- 스케줄러는 매일 `DAILY_PUBLISH_TARGET`(기본 5)건만 큐에 올린다.
- 이미 `queued`+`publishing`인 양을 빼고 부족분만 채운다(중복 큐잉 방지).
- 각 글의 `next_run_at`을 `PUBLISH_SPACING_MIN`(기본 90분) 간격 + 지터로 흩뿌린다.
- 발행 워커는 `next_run_at <= now`인 것만 집어 실제 발행.
- (후속) 발행 허용 시간대(예 09~21시) 윈도우 적용.

### 3.3 재시도 / 격리 정책

| 실패 유형 | 처리 | 비고 |
|---|---|---|
| `RetryableError` | attempts+1, 지수 백오프(1→2→4분…최대 1시간)+지터 후 `queued` | 일시적 차단·타임아웃·5xx |
| attempts ≥ `max_attempts`(기본 4) | `needs_human` + 알림 | 수동 폴백 큐 |
| `FatalError` | 즉시 `failed` + 알림 | 인증 영구실패·정책 위반 |
| 예상치 못한 예외 | 재시도 대상으로 처리 | 보수적 |

멱등성: `idem_key`(채널+제목+id 해시, UNIQUE)로 재시도가 중복 발행하지 않게.
원자성: `claim(queued→publishing)`로 워커 동시 실행 시 한쪽만 소유.

### 3.4 운영 cron (단계 격리)

```
*/30 * * * *  python run.py generate
*/30 * * * *  python run.py review
0    9 * * *  python run.py schedule
*/5  * * * *  python run.py publish
```

### 3.5 모니터링 지표

| 지표 | 신호 | 대응 |
|---|---|---|
| `reviewed` 재고량 | 버퍼 이하 | 생성량↑ / 시드 보충 |
| 검수 통과율 | 하락 | 모델 등급/프롬프트 점검(02) |
| `needs_human` 누적 | 증가 추세 | 채널 셀렉터·세션 점검(04) |
| 채널별 발행 성공률 | 특정 채널 급락 | 해당 어댑터 수선 |

## 4. 골격 반영 사항

- `config.py`: 발행 시간대 윈도우 변수(`PUBLISH_WINDOW_START/END`) 추가(후속 적용).
- `pipeline/publish.py`: `_notify()`를 슬랙/이메일에 연결(후속).
- 상태별 건수 대시보드/리포트 `tools/status_report.py`(후속).
