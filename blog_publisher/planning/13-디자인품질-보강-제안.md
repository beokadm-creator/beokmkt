## 0. 한 줄 결론

"텍스트만 길고 디자인이 약하다"의 **진짜 원인은 원고가 아니라 렌더링·이미지 레이어**다.
최근 발행 원고(id 121·94)는 이미 H2 8 / **H3 23~24** / 굵게 84~99 / 표 2~3 / 인용 4 / 이미지 3~8로
**마크업이 충분히 풍부**한데, ① CSS가 밋밋해 그 구조가 시각적으로 안 살고, ② 렌더러에 깨짐 결함이 있고,
③ 이미지가 작은 고정 스톡을 전 글에 돌려쓴다. 세 군데를 고치면 같은 원고가 전혀 다른 품질로 보인다.

## 1. 진단 — 실제 산출물에서 확인된 사실

### 1.1 원고 마크업은 이미 충분하다 (원고 깊이 문제 아님)

| 글 | 글자수 | H2 | H3 | 굵게 | 표 | 인용 | 이미지 |
|---|---|---|---|---|---|---|---|
| id 121 (hong, published) | 11,934 | 8 | 23 | 99 | 2~3 | 4 | 8 |
| id 94 (beok, published) | 11,382 | 8 | 24 | 84 | 3 | 4 | 3 |

→ 생성 프롬프트(`SECTION_SYSTEM`)는 굵게·표·H3·콜아웃을 이미 지시하고, 모델도 따르고 있다.
**보강 포인트는 "더 쓰게" 가 아니라 "있는 구조를 잘 보이게"** 다.

### 1.2 렌더러 결함 — 산출물에 그대로 노출 (P0)

`render/renderer.py`로 id 121을 실제 렌더한 결과(패치 전):

- **제목 줄에 붙은 이미지가 깨진다.** 원고가 `## 결론 요약 … 것이다![alt](url)` 처럼 H2 끝에 이미지를
  붙여 쓰는데, 변환기가 이미지를 `<h2>` **안에** 넣어 버린다 → 제목 옆에 이미지가 끼는 레이아웃 붕괴.
- **목차·요약 카드에 마크다운 원문이 그대로 노출된다.** 위 제목이 목차/요약 불릿에
  `…것이다![홍커뮤니케이션 e-Regi 스마트 행사 등록 시스템 화면](https://hongcomm.kr/img/page/a1.png)`
  같은 **날 마크다운 텍스트로** 찍힌다. 독자 눈에 바로 보이는 품질 저하.
- **앵커(슬러그)도 이미지 문구를 포함**해 지저분하고 깨지기 쉽다.

### 1.3 CSS가 밋밋함 — 풍부한 마크업이 시각적으로 안 산다 (P1)

기존 `style.css`(255줄)는 단색 강조 1개, 시스템 폰트, 기본 여백 수준. H2/H3/표/굵게가 있어도
**회색 본문 덩어리**로 읽힌다. 섹션 번호·리듬·콜아웃 변형·표 디자인·히어로 같은 시각 위계 장치가 없다.

### 1.4 이미지가 작고 반복됨 — "퀄리티 저하"의 큰 축 (P1)

- 자체 블로그 `featured_image`는 `tools/image_bank.py`의 **고정 풀**(hongcomm 480×300 썸네일 + 브랜드 SVG 4종)에서
  골라 **모든 글에 돌려쓴다.** 네이버·구글 모두 **원본 이미지**를 품질·고유성 신호로 보는데 불리하고,
  시각적으로도 글마다 같은 그림이 반복된다.
- 글별 고유 대표 이미지(OG/히어로)가 없다. (SEO 로드맵 3.4와 동일 진단)

### 1.5 채널별 현황 — 차별화 골격은 있으나 점검 필요

| 채널 | 디자인 경로 | 현황 | 남은 일 |
|---|---|---|---|
| 자체 블로그 | `render/renderer.py` + `style.css` | **이번 패치로 대폭 보강** | OG카드 연동·배포 |
| 티스토리 | `channel-rewriter.mjs` → `tistory-html-adapter.mjs`(인라인 스타일) | **이미 양호**: 재작성+인라인 디자인 존재(`CHANNEL_REWRITE="true"`) | 재작성 차별도(70%+)·고유 이미지 |
| 네이버 | `convertForNaver`(리치 제거) | 천장 낮음(구조·이미지까지) | 유지 |

> 주의: `server/blog-pipeline/html-templates.mjs`는 **Tailwind 클래스(zinc-…)** 로 배너/CTA를 감싼다.
> SPA(자체 블로그 React) 안에서만 스타일이 먹고, 그 HTML을 외부(RSS/티스토리)로 그대로 내보내면 무스타일이 된다.
> 자체 블로그 표준 렌더는 `render/`(인라인 CSS) 한 곳으로 모으는 것을 권장(이중 경로 정리).

## 2. 적용한 패치 (검증 완료)

실제 발행 원고(id 121·94)를 패치 전/후로 렌더해 사람이 눈으로 확인했다. (산출물 HTML 동봉)

### 2.1 `render/renderer.py` (+89줄, 하위호환·가산형)

| 추가/수정 | 효과 |
|---|---|
| `_clean_heading_text()` | 제목/목차/요약에서 마크다운(이미지·링크·굵게·코드) 제거 → **원문 노출 결함 차단** |
| `_normalize_block_images()` | 제목 줄에 붙은 이미지를 **독립 블록으로 분리** → `<h2>` 안 이미지 끼임 제거 |
| H2 id 순서 부여 방식 | 내부 태그/강조가 있어도 앵커가 안전하게 매칭 |
| `<figure>`+`<figcaption>` 승격 | 단독 이미지를 캡션 있는 그림 카드로 → 가독성/시각 위계 ↑ |
| 콜아웃 변형(info/tip/warn) | 머리말 키워드(주의/팁 등)로 색·아이콘 자동 분기 |
| `_hero_html()` | `og_image`/`hero_image`가 있으면 본문 상단 **히어로 이미지** 노출(없으면 미표시=무회귀) |

**검증(패치 전 → 후, id 121):** 목차/요약 마크다운 원문 노출 `있음 → 없음`,
`<h2>` 내부 이미지 `있음 → 없음`, `<figure>` `0 → 8`, 표 래퍼 5, 콜아웃 유지.
`markdown` 패키지 유무(경량 폴백) **양쪽 경로 모두** 동일하게 통과.

### 2.2 `render/style.css` (전면 보강, 클래스명 100% 유지)

시각 위계를 만드는 디자인 시스템으로 교체:

- **번호 칩 H2**(01·02…), 강조 밑줄형 굵게, 큰 리드 문단, H3 좌측 액센트 바.
- **그림 카드**(테두리·그림자·캡션), **표 디자인**(헤더 액센트·짝수행 음영·라운드 래퍼·가로 스크롤).
- **콜아웃 3변형**(info/tip/warn 아이콘), **핵심요약 카드**(헤더 바), **목차 카드**(번호), 태그 pill, CTA 그라데이션 카드.
- 다크모드·모바일 반응형·`word-break:keep-all`(한국어 줄바꿈) 정리.

### 2.3 `tools/og_card.py` (신규) — 글마다 고유한 대표 이미지

- 외부 의존 0. **제목·카테고리·키워드만으로 글마다 다른 1200×630 SVG 카드**를 결정론적으로 생성
  (카테고리별 팔레트 + 제목 해시로 색조/패턴 변주). `build_og_svg(post)` / `save_og_card(post, dir)`.
- 용도: `featured_image`(OG/소셜 카드) + 본문 상단 히어로. **고정 스톡 반복 문제를 직접 해소.**
- 실제 제품 스크린샷이 확보되면 그쪽을 우선하고 OG카드는 폴백으로 두면 된다(품질 North-Star §C와 일관).

## 3. 제안 패치 (선택 필요 — 호스팅 결정 후 적용)

### 3.1 OG카드를 자체 블로그 발행에 연동 (`publishers/selfhosted.py`)

현재 `featured_image()`(고정 풀) 대신/보완으로 글별 OG카드를 쓰려면, **SVG를 URL로 호스팅**해야 한다.
두 가지 경로 중 택1:

```python
# publishers/selfhosted.py — featured_image 직전
from tools.og_card import build_og_svg, save_og_card

# (A) 호스팅형: public/assets/blog/og/<slug>.svg 로 저장 → 배포 파이프라인이 업로드
card_path = save_og_card(post, "public/assets/blog/og")
og_url = f"https://beokmkt.web.app/assets/blog/og/{card_path.name}"

# featured_image 우선순위: 실제 제품 이미지 > OG카드 > 고정 풀
image_url = (featured_image(...).get("url")) or og_url

payload["featured_image"] = image_url
# 본문 상단 히어로도 같은 이미지로:
content = render_body({ ..., "hero_image": image_url })
```

> 결정 필요: (A) 정적 파일 호스팅(배포 1줄 추가) vs (B) 셀프호스트 API에 이미지 업로드 엔드포인트가 있으면 그걸 사용.
> 호스팅 경로만 정해지면 즉시 적용 가능. (데이터-URI는 OG/소셜 미리보기에서 제대로 안 떠서 비권장.)

### 3.2 채널 차별화 — 이미 있는 재작성을 "충분히 다르게"로

- `CHANNEL_REWRITE="true"`라 티스토리/네이버는 재작성이 돈다. **차별도 측정**(원본 대비 n-gram 중복률)을
  로그에 남기고, 70% 미만이면 재작성 강도를 높이도록 `channel-rewriter.mjs` 가이드에 임계값을 명시.
- **채널마다 다른 OG카드 색/문구**를 쓰면 이미지 단계에서도 유사문서 신호를 줄일 수 있다(같은 `og_card`에 채널 인자만 추가).

### 3.3 렌더 경로 일원화

자체 블로그 표준 렌더를 `render/`(인라인 CSS)로 통일하고, `server/blog-pipeline/html-templates.mjs`의
Tailwind 의존 배너/CTA는 SPA 전용임을 주석으로 못 박거나 인라인 스타일로 교체.

## 5. 검증 방법 (품질 North-Star §0 준수: 산출물을 직접 읽는다)

1. `before_*.html` / `after_*.html`을 브라우저에서 열어 PC·모바일로 직접 비교.
2. 신규 1~2건을 실제 생성→렌더해 목차/요약에 마크다운 원문이 없는지, 히어로/표/콜아웃이 보이는지 확인.
3. OG카드는 `python -c "from tools.og_card import save_og_card"`로 몇 건 뽑아 색/문구 다양성 확인.
4. 발행 연동(§3.1) 후 공개 URL에서 `featured_image`·OG 미리보기(카카오/구글)까지 확인.
