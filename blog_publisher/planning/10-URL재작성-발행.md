- 상태: 확정
- 작성일: 2026-06-14
- 관련 골격: `pipeline/rewrite.py`, `research/extract.py`, `utils/similarity.py`, `config.py`, `db/schema.sql`
- 선행 문서: [05](./05-근거기반-콘텐츠엔진-재설계.md), [07](./07-검색노출-SEO-최적화.md)

## 1. 배경 / 중요한 경고

"다른 블로그 URL을 넣으면 그 내용을 재작성해 발행" 기능. 단, **그대로 두면 위험**하다:

1. **저작권 침해**: 타인 글을 30%만 바꿔 발행하는 것은 2차적저작물작성권 침해 소지가 크다.
2. **중복 콘텐츠 패널티**: 구글·네이버 모두 스핀/복제 콘텐츠를 강등·누락시킨다. 노출 목적과 정면 배치된다.

따라서 "30%만 바꾸기"가 아니라 **30%를 최소 변경 하한**으로 두되, 실제 설계는:
- 원문을 **하나의 참고자료**로만 사용(여러 출처 중 하나로 취급, 가능하면 추가 리서치 병행),
- **대폭 재구성 + 독창적 분석/정리 추가**,
- **출처 표기**(rel=nofollow 링크),
- **유사도 검증**으로 원문과 충분히 달라졌는지 자동 확인.

이게 법적으로도 노출 측면에서도 안전한 유일한 방향이다.

## 2. 목표

- 입력: 원문 URL(+유형/채널/locale).
- 출력: 원문과 충분히 다른(유사도 임계 미만), 출처 표기된, 독창 가치가 더해진 새 글.
- 안전장치: 유사도 게이트 + 출처 표기 + (옵션) 추가 리서치.

## 3. 설계 / 결정

### 3.1 파이프라인

```
원문 URL
 → ① 본문 추출 (extract)
 → ② 핵심 사실/주장 요약 (LLM, 근거팩 facts와 동일 포맷·원문 출처 귀속)
 → ③ (옵션) 추가 웹 리서치로 근거 보강 (05 재사용)
 → ④ 재구성 작성 : 새 관점/구조/제목으로 합성(원문 문장 복붙 금지)
 → ⑤ 유사도 검증 : 원문 대비 유사도 < MAX_SIMILARITY 이어야 통과
       실패 시 재작성(더 강한 변형 지시) 또는 보류
 → ⑥ 출처 표기 부착
 → ⑦ 사실검증/품질검수/SEO (기존 게이트 재사용)
 → 발행
```

### 3.2 유사도 검증

- 1차(무료): 어절 n-gram 자카드/중복률(`utils/similarity`) — 원문과 초안 비교.
- `MAX_SIMILARITY`(기본 0.3) 이상이면 탈락 → 재작성. (즉 "최소 30% 변경"이 아니라 "70% 이상 달라야 통과"로 강하게.)
- (후속) 임베딩 코사인 유사도로 의미 유사까지 점검.

### 3.3 출처 표기

- 본문 하단 "참고 출처" 링크(렌더러가 `rel=nofollow`로 출력, 기획 09 연동).
- DB `source_url` 저장.

## 4. 골격 반영

- `research/extract.py`: URL 본문 추출(공급자 fetch + 간이 정제, readability 있으면 사용).
- `utils/similarity.py`: n-gram 자카드 유사도.
- `pipeline/rewrite.py`: 위 파이프라인. `rewrite_from_url(url, content_type, channel)`.
- `db/schema.sql`: `source_url`(이미 추가). 시드에 source_url 경로.
- `config.py`: `MAX_SIMILARITY`, 재작성 재시도 횟수.
- `run.py`: `rewrite <url> <channel> <type>` 명령.

## 5. 미해결 / 다음 단계

- 임베딩 기반 의미 유사도(표면만 바꾼 스핀 차단).
- robots.txt/이용약관 준수, 스크래핑 차단 도메인 예외.
- 인용 비율 정책(짧은 직접 인용은 출처 명시 후 허용 범위).
- 원문이 부실하면 거부(저품질 재생산 방지).
